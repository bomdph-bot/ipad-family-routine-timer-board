     1|# iPad 家庭晚间任务倒计时奖励面板 — 首版调研
     2|
     3|- Idea ID: `ipad-family-routine-timer-board`
     4|- 对应 Issue: #8
     5|- 当前阶段产物：Research v2
     6|- 本版重点：吸收用户在 PR 中的新反馈，把 MVP 从“单孩子晚间面板”修正为“4 个孩子可切换 + 家长后台 + 可配置时间窗口”的家庭任务时间面板，同时保留首版应避免范围失控的边界
     7|
     8|## 1. 问题定义
     9|
    10|这个构想要解决的核心问题，不是“再做一个待办清单”，而是让孩子能在放学后的固定 3 小时里，直观看见时间流动、任务占用与提前完成后的奖励，从而减少拖拉、磨蹭和家长口头催促。
    11|
    12|当前痛点主要有 4 类：
    13|
    14|1. 时间感抽象
    15|   - 孩子往往知道“今晚要做很多事”，但对 6pm 到 9pm 这段时间怎么被切分、哪些任务正在挤占自由时间，没有直观感受。
    16|
    17|2. 家长催促成本高
    18|   - 家长需要不断提醒洗澡、吃饭、作业、运动、洗漱等事项，容易变成情绪对抗。
    19|
    20|3. 完成任务缺少即时反馈
    21|   - 普通清单只会显示“完成/未完成”，但不能把“你提前做完，换来了更多自由时间”可视化。
    22|
    23|4. 任务先后顺序与剩余时间压力不透明
    24|   - 当剩余时间小于未完成任务总时长时，孩子不会自然意识到“已经来不及了”，需要一个会自己表达紧迫感的界面。
    25|
    26|因此，这个产品的本质是一个“以时间为主轴的家庭晚间节奏面板”，重点是让孩子看懂：
    27|
    28|- 现在剩多少时间
    29|- 还剩哪些任务
    30|- 哪些任务已经完成
    31|- 自己做得快，会换来多少可自由支配时间
    32|- 做得慢，会让剩余任务越来越拥挤、越来越紧迫
    33|
    34|## 2. 目标用户与核心场景
    35|
    36|### 2.1 目标用户
    37|
    38|1. 家长：负责在后台为 4 个孩子配置任务、时长、共用任务与个人独立任务，并维护每日时间窗口与规则
    39|2. 孩子：在 iPad 上查看自己的当前面板、按顺序完成任务、理解下一步要做什么，并感知完成效率带来的自由时间
    40|
    41|### 2.2 核心场景
    42|
    43|#### 场景 A：每天放学后打开 iPad 面板
    44|
    45|- 孩子看到 6:00pm 到 9:00pm 的整体倒计时条
    46|- 时间条里按顺序放着不同颜色的任务格子
    47|- 每个格子长度代表预计时长
    48|- 左侧随着时间消耗而缩短，孩子能持续感知“今晚还剩多少时间”
    49|
    50|#### 场景 B：完成一个任务后立即获得奖励反馈
    51|
    52|- 例如先完成洗澡 20 分钟
    53|- 对应任务格子被点击完成后消失或转为完成状态
    54|- 如果完成效率高，左侧会明显出现一段蓝色荧光自由时间
    55|- 孩子能直观看到“我快一点，就多出可玩时间”
    56|
    57|#### 场景 C：时间开始不够用时，界面主动制造紧迫感
    58|
    59|- 如果剩余时间已经小于未完成任务总时长
    60|- 未完成任务格子出现被挤压、发热、膨胀等视觉反馈
    61|- 不需要家长一直说“快点”，界面本身就会传达压力
    62|
    63|#### 场景 D：四个孩子切换与共用 / 独立任务并存
    64|
    65|- 家里有四个孩子，切换功能不是可选项，而是第一版就要有
    66|- 系统里既有所有孩子共用的任务，也有每个孩子自己独立的任务
    67|- 家长需要能在后台区分“共用任务模板”和“个人专属任务”
    68|- 孩子在前台打开时，应能清楚看到自己当前应该做的下一步，而不是被兄弟姐妹的任务干扰
    69|
    70|#### 场景 E：家长后台设置每日节奏
    71|
    72|- 家长需要进入后台，为不同孩子设置任务名称、预计时长、顺序与是否共用
    73|- 家长还需要能调整当天的起始时间与整体时间窗口，例如有时 5:30pm 就到家，不一定总是 6:00pm 开始
    74|- 后台应服务于“快速配置今晚安排”，而不是做成复杂表格系统
    75|
    76|#### 场景 F：非 6pm-9pm 时段打开界面
    77|
    78|- 如果比计划开始时间更早打开，界面不应报错或空白，而应展示“今晚计划尚未开始、即将开始于 xx:xx”的待机状态
    79|- 如果已经进入计划时间窗口，显示实时倒计时与任务格子
    80|- 如果超过计划结束时间但任务未做完，应进入“超时 / 延后”视图，明确剩余未完成任务，而不是直接消失
    81|- 这说明产品底层不应把 6pm-9pm 写死成唯一规则，而应允许存在一个默认模板和当天可调整的时间范围
    82|
    83|## 3. 为什么值得做
    84|
    85|1. 它把“催促”转换为“可视化反馈”
    86|   - 家长不需要一直靠嘴提醒，系统自己用时间条和任务格子表达进度。
    87|
    88|2. 它把“完成任务”与“得到奖励”直接绑定
    89|   - 自由时间不是抽象说教，而是会在界面上立刻长出来。
    90|
    91|3. 它有明确家庭教育价值
    92|   - 帮助孩子建立时间观念、顺序意识、完成感与效率习惯。
    93|
    94|4. 它天然适合 iPad 场景
    95|   - 大屏、可触摸、固定地点展示，非常适合当晚间家庭节奏看板。
    96|
    97|5. 它也适合作为后续家庭规则系统的入口
    98|   - 后续如果要加家长设置页、奖励机制、值日轮换、统计报表，这个时间面板就是最自然的主界面。
    99|
   100|## 4. 参考方案 / 开源参考
   101|
   102|以下参考不是直接照搬，而是用于验证“时间可视化 + 时间轴排布 + PWA 形态”这条方向是现实可行的。
   103|
   104|### 4.1 task-timer-pwa
   105|
   106|- 项目：`JAKimball/task-timer-pwa`
   107|- 链接：https://github.com/JAKimball/task-timer-pwa
   108|- 描述：Visual task timer for corner-of-the-eye time awareness.
   109|
   110|可借鉴点：
   111|- 任务计时器做成始终可感知的视觉对象是成立的
   112|- “不用频繁盯数字，也能感受时间压力”这个思路与你的目标一致
   113|
   114|不足：
   115|- 更偏个人任务计时，不是面向孩子的家庭晚间节奏面板
   116|- 没有你想要的任务格子奖励逻辑、多孩子切换和家长规则设置
   117|
   118|### 4.2 react-calendar-timeline
   119|
   120|- 项目：`namespace-ee/react-calendar-timeline`
   121|- 链接：https://github.com/namespace-ee/react-calendar-timeline
   122|- 描述：A modern and responsive react timeline component.
   123|
   124|可借鉴点：
   125|- 说明“按时间轴摆放多个区块”的交互在前端实现上是成熟路线
   126|- 可以参考其时间条、区块布局、缩放与渲染方式来验证任务格子方案
   127|
   128|不足：
   129|- 它更像通用时间轴组件，视觉气质和交互目标偏业务工具
   130|- 不适合直接拿来当儿童向最终 UI，只适合作为实现思路参考

   132|## 5. 可行性判断
   133|
   134|### 5.1 UI 原型层：高可行
   135|
   136|这个构想最强的部分是画面感已经很清楚：
   137|
   138|- 固定 3 小时主时间条
   139|- 不同颜色任务格子
   140|- 倒计时实时缩短
   141|- 任务完成后格子消灭
   142|- 自由时间发光
   143|- 超时压力时格子挤压/膨胀
   144|
   145|这意味着第一阶段完全可以先做高保真 UI 原型和交互规则，不需要一开始就做复杂后端。
   146|
   147|### 5.2 iPad PWA 层：高可行
   148|
   149|- iPad Safari 对 PWA 支持已经足够支撑面板式应用
   150|- 这种产品主要是单页、触控、定时刷新、状态显示，非常适合 PWA
   151|- 后续家长设置页与孩子展示页也可以在同一 Web 应用里完成
   152|
   153|### 5.3 NAS Docker 部署层：中高可行
   154|
   155|- 如果先做前端为主、数据量不大的版本，用 NAS Docker 部署完全合理
   156|- 后续若需要多孩子数据、每日模板、历史统计，可再加轻量后端与数据库
   157|
   158|### 5.4 “防小孩乱点”机制：可行，但第一版先做轻量纠偏
   159|
   160|你提到的重点已经从“要不要值日小管家”进一步明确成：
   161|
   162|- 第一版必须有 4 个孩子切换
   163|- 必须有家长后台去配置任务和时长
   164|- 系统的目标之一，是让孩子在空闲时也能自己抬头看面板，知道下一步该做什么
   165|
   166|这意味着第一版真正优先级更高的，不是引入复杂家庭治理角色，而是先把“谁的面板、今晚要做什么、何时开始、当前下一步是什么”做清楚。
   167|
   168|因此更稳的默认推进方案是：
   169|
   170|1. MVP 先做轻量防误触 / 防作弊机制
   171|   - 完成任务时弹出二次确认
   172|   - 或允许家长在后台撤销最近一次完成
   173|   - 必要时再加“家长确认模式”开关
   174|
   175|2. 先不把“值日小管家制度”做成正式系统角色
   176|   - 可以在 PRD 里保留为后续扩展思路
   177|   - 但不让它抢占 4 孩子切换、家长后台、时间窗口配置这些更核心的 MVP 资源
   178|
   179|## 6. 建议的 MVP 收敛方式
   180|
   181|我建议第一版不要再收敛成“单孩子演示版”，而是直接做“4 个孩子切换 + 家长后台 + 可配置晚间时间窗口”的最小真实闭环。
   182|
   183|### 6.1 MVP 目标
   184|
   185|让家长可以为四个孩子配置今晚任务，让孩子在 iPad 上切换到自己的任务面板，看到当前时间窗口内的任务条、未被任务占据而发光的自由时间，以及下一步该做什么。
   186|
   187|### 6.2 MVP 范围内
   188|
   189|1. 支持 4 个孩子切换入口
   190|2. 家长后台最小版本：编辑孩子、任务名称、顺序、时长、是否共用、当日起始时间 / 结束时间
   191|3. 支持“共用任务 + 个人独立任务”组合生成每个孩子自己的面板
   192|4. 默认存在一个晚间模板，但允许家长把时间窗口从固定 6:00pm-9:00pm 调整为例如 5:30pm-8:30pm
   193|5. 按顺序排列的任务格子（洗澡、吃饭、作业、运动、洗漱等）
   194|6. 每个任务格子按预计时长显示长度
   195|7. 实时倒计时与时间条缩短
   196|8. 点击任务完成后更新界面
   197|9. 所有未被任务格子占据的时间，以明显发光的自由时间区域呈现
   198|10. 剩余时间不足时，对未完成任务显示紧迫视觉效果
   199|11. 计划开始前、进行中、结束后未完成三种基础状态视图
   200|
   201|### 6.3 明确非目标
   202|
   203|第一版先不追求：
   204|
   205|- 正式的值日小管家角色制度
   206|- 自动奖励积分系统
   207|- 语音提醒 / 硬件联动
   208|- 长周期统计报表
   209|- AI 自动排任务
   210|- 精细权限体系
   211|- 云端多端同步的复杂账号系统
   212|- 复杂周计划 / 月计划编排
   213|
   214|## 7. 关键设计判断
   215|
   216|### 7.1 主视觉重心应该是“时间条”，不是“清单”
   217|
   218|因为这个产品最有差异化的地方是把时间直接物化成可消耗、可争取、可被任务挤压的对象。
   219|
   220|### 7.2 自由时间反馈应理解为“未被任务占据的发光时间”
   221|
   222|用户已经补充得很明确：自由时间不一定要绑定复杂奖励规则，它首先是时间条上“未被任务格子占据的部分”。
   223|
   224|也就是说：
   225|
   226|- 先按预计时长给任务格子占位
   227|- 如果孩子更快完成任务，任务格子被消除后，剩余未被占据的时间要明显发光
   228|- 这个发光区域本质上是在告诉孩子：这些是你把任务做完后换出来的自由活动时间
   229|
   230|因此第一版的重点不是做一套复杂积分规则，而是把“空出来的时间 = 看得见的自由时间”表达得足够清楚。
   231|
   232|### 7.3 紧迫感反馈要可爱但不能太吓人
   233|
   234|“发热膨胀化”很有画面，但儿童向 UI 需要控制强度。建议偏可爱、轻微夸张，而不是红色警报式压迫界面。
   235|
   236|### 7.4 家长模式与孩子模式应尽量分离
   237|
   238|孩子页面应该专注执行与反馈，家长页面负责设置与纠错。不要把复杂操作塞进孩子主界面。
   239|
   240|## 8. 待确认边界
   241|
   242|1. 四个孩子切换 + 家长后台 + 共用 / 独立任务已确定进入 MVP，但具体信息架构还需在 PRD 里收敛
   243|2. 任务完成是否允许孩子自己点，还是默认允许点完后再由家长撤销纠偏
   244|3. 计划开始前的待机界面、计划结束后的超时界面，默认文案和视觉风格要做到什么程度
   245|4. 是否需要把“当前下一步任务”做成比普通格子更强的高亮提示
   246|5. 多孩子切换入口是顶部头像切换就够，还是需要单独的角色主页
   247|
   248|## 9. 推荐下一步
   249|
   250|建议下一阶段可以进入 PRD，但 PRD 不应再按旧的“单孩子晚间条”去写，而应优先回答以下问题：
   251|
   252|1. 4 个孩子的切换入口与主页信息层级怎么设计
   253|2. 家长后台如何设置共用任务、个人任务、起止时间与顺序
   254|3. 孩子主界面如何突出“当前下一步任务”
   255|4. 计划开始前、进行中、结束后未完成，这三种状态分别展示什么
   256|5. 任务完成的最轻纠偏方案采用哪一种（允许孩子点、二次确认、家长撤销、家长确认模式）
   257|6. MVP 第一版是先做高保真前端原型，还是直接接上 NAS 上的轻量数据存储
