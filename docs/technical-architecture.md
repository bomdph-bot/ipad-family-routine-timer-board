---

## 1. 技术选型建议

### 1.1 前端：React + Vite + PWA

**选型理由：**

| 备选方案 | 结论 | 原因 |
|---------|------|------|
| React + Vite | ✅ 推荐 | 生态成熟，组件复用高，Vite 热更新快，适合快速迭代 |
| Vue | ❌ 不选 | 与 React 二选一，本项目 React 社区资源更丰富 |
| 纯 HTML/JS | ❌ 不选 | 无组件化，任务格子状态管理会快速失控 |
| Flutter / RN | ❌ 不选 | iPad 专属场景不需要跨平台开销，PWA 更轻 |

**PWA 选型：**
- 使用 `vite-plugin-pwa` 生成 Service Worker
- 支持离线缓存（局域网 iPad 使用场景，稳定性优先）
- 可添加到 iPad 主屏幕，像原生 App 一样运行
- iPad Safari PWA 支持足够成熟，面板类单页应用无需复杂推送

**UI 框架：**
- 使用 TailwindCSS 做快速样式开发
- 自定义任务格子组件（时间条核心可视化，不依赖第三方 Timeline 库）

---

### 1.2 后端：Node.js + Fastify（轻量路线）

**选型理由：**

| 备选方案 | 结论 | 原因 |
|---------|------|------|
| Node.js + Fastify | ✅ 推荐 | 轻量、JSON 处理快、学习成本低、适合家庭轻量场景 |
| Python Flask | ❌ 不选 | 同样轻量，但 Node.js 与前端同构更好维护 |
| Go | ❌ 不选 | 对于第一版数据量，复杂度不必要 |
| PHP | ❌ 不选 | 维护性和现代开发体验不佳 |

**Fastify 优势：**
- 默认 JSON 序列化快
- 插件生态丰富
- 内存占用低，适合 NAS 环境

**初期可省后端：** 第一版原型阶段前端可使用 localStorage 模拟数据，验证 UI 和交互后再接入后端 API。

---

### 1.3 存储：SQLite + 简单文件备份

**选型理由：**

| 备选方案 | 结论 | 原因 |
|---------|------|------|
| SQLite | ✅ 推荐 | 零配置、单文件、无服务进程、NAS 友好 |
| PostgreSQL | ❌ 不选 | 需要独立数据库服务，NAS 资源有限 |
| MySQL | ❌ 不选 | 同上， heavier |
| MongoDB | ❌ 不选 | 对本项目结构化数据场景过于重量 |
| localStorage | ⚠️ 仅原型 | 原型阶段可用，正式版换 SQLite |

**数据备份：**
- SQLite 文件可通过 NAS 文件服务直接备份
- 可定期将 `.db` 文件复制到 NAS 共享目录

**扩展路径：**
- 若后续需多设备同步，可引入 `better-sqlite3` 或 `sqlite-wasm`（浏览器端直连）
- 若需远程访问，可加轻量 REST API，不引入 ORM 复杂度

---

### 1.4 部署：Docker + Docker Compose（NAS）

**选型理由：**

| 备选方案 | 结论 | 原因 |
|---------|------|------|
| Docker on NAS | ✅ 推荐 | 群晖/威联通等主流 NAS 均支持，隔离性好，迁移方便 |
| 直接裸机部署 | ❌ 不选 | 不便于版本管理和回滚 |
| Kubernetes | ❌ 不选 | 家庭 NAS 场景过于复杂 |
| 纯前端静态托管 | ⚠️ 仅原型 | 第一版无后台可这样跑，但后续需要状态持久化 |

**Docker 镜像规划（单容器）：**
```
镜像名：ipad-timer-backend
内容：Node.js + Fastify + SQLite
端口：3000
数据卷：./data:/app/data（SQLite 文件持久化）
```

**iPad 访问路径：**
- 局域网直接访问 `http://<NAS-IP>:3000`
- 可通过 NAS 内置反向代理绑定域名，无需暴露端口

## 2. 系统边界与模块拆分

### 2.1 模块边界图（文字版）

                    ▼
         ┌──────────────────┐

### 2.2 模块职责

| 模块 | 职责 | 边界 |
|------|------|------|
| 孩子主面板 | 展示当前孩子任务条、时间倒计时、下一步任务、自由时间、紧迫感反馈 | 只读/交互，不做配置 |
| 家长后台 | 配置孩子、任务、时长、顺序、共用/独立、时间窗口 | 只写配置，不做孩子界面操作 |
| 家长身份验证 | 简单 PIN 码验证，防止孩子进入后台乱改 | 非完整账号系统，第一版极简 |
| REST API | 提供数据读写接口，连接前端与 SQLite | JSON API，不做复杂业务逻辑 |
| 数据层 | SQLite 存储孩子、任务、计划数据 | 单文件，支持备份 |
| NAS Docker | 提供容器化运行环境和数据持久化 | 群晖/威联通通用 |

### 2.3 数据模型（简化版）

```
Children 表
├── id
├── name
├── color
└── sort_order

Tasks 表
├── id
├── title
├── duration_minutes
├── order_index
├── scope_type (shared / personal)
├── assigned_child_id (共用时为 null)
└── is_template (模板任务，供每日复制用)

Plans 表
├── id
├── plan_date
├── start_time
└── end_time

PlanTasks 表（每日计划与任务的关联）
├── plan_id
├── task_id
├── child_id (哪个孩子的任务)
└── status (pending / done / skipped)
```

**注意：** 第一版不设计多设备同步账号体系，数据存储以本地 SQLite 为主。

## 3. 实施阶段建议

### 阶段 1：原型验证（UI 先行）

**目标：** 验证核心交互和视觉反馈，暂不碰后端

**产出：**
- 高保真 Figma 原型 或 HTML 原型（可点击跳转）
- 任务格子时间条交互原型
- 紧迫感/自由时间视觉反馈验证

**技术栈：**
- 纯 HTML + CSS + JS 或 React + localStorage
- 模拟 4 个孩子数据

**交付物：**
- `prototype.md` 或 Figma 链接
- 评审通过后进入阶段 2

**判断标准：**
- 孩子主面板时间条视觉是否符合预期
- 紧迫感反馈是否达到"可爱但不吓人"标准
- 自由时间发光效果是否足够直观

---

### 阶段 2：前端工程化（React PWA）

**目标：** 搭建可安装到 iPad 主屏幕的 PWA

**产出：**
- React + Vite 项目骨架
- PWA Service Worker 配置（离线可用）
- 孩子主面板完整实现
- 家长后台最小配置页
- 简单 PIN 码验证

**技术栈：**
- React 18
- Vite + vite-plugin-pwa
- TailwindCSS
- Zustand（轻量状态管理）

**交付物：**
- GitHub 前端仓库（`/frontend` 目录）
- 可在 localhost 运行的完整 PWA
- 安装到 iPad 的测试报告

**判断标准：**
- PWA 可添加到 iPad 主屏幕
- 4 个孩子切换正常
- 家长 PIN 码可正常保护后台

---

### 阶段 3：数据层（SQLite + Fastify API）

**目标：** 将前端数据从 localStorage 迁移到 SQLite，建立 REST API

**产出：**
- Fastify 后端服务
- SQLite 数据库初始化脚本
- 基础 CRUD API
- Docker + Docker Compose 配置
- 数据迁移脚本（从 localStorage 导入）

**技术栈：**
- Node.js 18+
- Fastify
- better-sqlite3
- Docker + Docker Compose

**交付物：**
- `/backend` 目录（与前端同仓库或独立）
- `docker-compose.yml`
- NAS 上可运行的 Docker 镜像
- API 文档（Swagger 或 README）

**判断标准：**
- 后端可在 NAS 上 Docker 启动
- 前端可连接 API 读写数据
- 数据重启后不丢失

---

### 阶段 4：NAS 部署与优化

**目标：** 完成生产级别部署，验证家庭真实使用

**产出：**
- NAS Docker 部署文档
- 数据备份方案（定时复制 db 文件）
- 局域网域名访问配置（可选）
- 性能优化（SQLite 索引、API 缓存）

**技术栈：**
- 群晖 Docker 套件（或威联通 Container Station）
- Nginx（NAS 反向代理，可选）

**交付物：**
- 部署 README（针对群晖小白用户）
- 一键启动脚本或 Docker Compose
- 使用验证报告

**判断标准：**
- 家长可在 NAS 后台完成所有配置
- 孩子可在 iPad 上流畅使用
- 数据可正常备份与恢复

## 4. 技术路线决策总结

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 前端框架 | React + Vite | 生态成熟，组件化好，适合有状态的面板应用 |
| PWA | 是 | iPad 无需应用商店分发，直接安装，离线可用 |
| 后端框架 | Fastify | 轻量高性能，JSON 处理快 |
| 数据库 | SQLite | 单文件零配置，NAS 友好，足够第一版 |
| 部署 | Docker on NAS | 隔离性好，版本管理清晰，家庭 NAS 支持好 |
| 账号系统 | 第一版不做 | 用简单 PIN 码代替，避免复杂账号体系拖累 |
| 多设备同步 | 第一版不做 | 局域网场景优先，同步留作后续 |

---

## 5. 下一步行动

1. **Phase 1 原型** — 可立即开始，用 HTML/CSS/JS 搭一个可点击的时间条原型
2. **Phase 2 前端** — 在原型验证后，搭建 React 工程并接入 PWA
3. **Phase 3 数据层** — 在前端核心交互完成后，建立 Fastify + SQLite
4. **Phase 4 部署** — 在功能稳定后，编写 Docker 配置和 NAS 部署文档

**建议顺序：Phase 1 → Phase 2 → Phase 3 → Phase 4**

MVP 验证路径：先让家长和孩子在 iPad 上看到可用的原型，再逐步完善数据持久化和部署。

---

## 6. 风险与备选

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| NAS Docker 性能不足 | 低 | 中 | SQLite 已足够轻量，第一版数据量极小 |
| iPad PWA 离线缓存失效 | 低 | 中 | Service Worker 配置后充分测试 |
| 孩子绕过 PIN 进入后台 | 低 | 中 | 第一版仅用于家庭内部信任场景 |
| 未来需要多设备同步 | 中 | 高 | API 已预留，SQLite 可迁移至云数据库 |
