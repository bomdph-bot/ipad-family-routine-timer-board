# OpenClaw 主 Agent 巡视 / 派工 / 兜底规则

## 1. 巡视目标

OpenClaw 的目标不是亲自做所有事，而是保证项目不会卡死：

- 有可推进的任务时，及时派发
- 任务不清晰时，先回到 Hermes 补拆分
- PR 有 review 意见时，优先回流到执行者
- 发现阻塞时，明确记录依赖关系和解锁条件

## 2. 巡视顺序

1. 先看项目立项 Issue
2. 再看当前 open 的阶段 / 子任务 Issue
3. 再看 open PR 的状态、检查结果与 review 结论
4. 最后判断是否需要用户拍板

## 3. 派工条件

一个任务只有同时满足以下条件，才适合派给子 Agent：

1. 输入文档明确
2. 目标单一
3. 依赖已满足
4. 验收标准可判断
5. 预计可由单个 PR 完成

## 4. 回流条件

出现以下情况时，不继续硬推，而是回流：

- Issue 目标描述含糊
- 依赖未满足
- PR review 未通过
- 任务明显超出当前范围
- 用户尚未对关键方向拍板

## 5. 对用户的最小打扰原则

只有在以下情况才应该把问题抛给用户：

- 产品方向存在真实分叉
- 当前问题无法通过已有文档判断
- 需要最终业务验收

其他情况优先在系统内部完成拆分、回流和修正。
